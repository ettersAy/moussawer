# New Chat Prompt: 401 Unauthorized on /api/v1/me

## Problem

After deploying to Render.com and migrating the database from SQLite to Supabase PostgreSQL, the frontend app makes a GET request to `/api/v1/me` and receives a **401 Unauthorized** response. The app cannot authenticate the user.

**Request details:**
- URL: `https://moussawer.onrender.com/api/v1/me`
- Method: GET
- Status: 401 Unauthorized
- Remote Address: `216.24.57.251:443`

## Root Causes (3 issues found)

### Root Cause 1: `render.yaml` uses `generateValue: true` for `JWT_SECRET`

**File:** `render.yaml`

```yaml
envVars:
  - key: JWT_SECRET
    generateValue: true   # ‚Üê BUG: generates a random secret on each deploy
```

This tells Render.com to **generate a random JWT_SECRET** at deploy time. This random secret is different from the local `.env` value (`d6fb73ec361ec5bd4bc60f8375e5f310`). Any JWT tokens stored in localStorage from development are signed with the local secret, so they fail verification on Render.com ‚Üí 401.

**Fix:** Use a hardcoded value matching the local `.env`:

```yaml
envVars:
  - key: JWT_SECRET
    value: d6fb73ec361ec5bd4bc60f8375e5f310
```

**Why this value?** The local `.env` file uses `d6fb73ec361ec5bd4bc60f8375e5f310`. Using the same value ensures tokens signed locally (during development) are valid on Render.com. If you change this, all existing users must log in again.

### Root Cause 2: `seed.sql` has invalid bcrypt hashes

**File:** `seed.sql`

The seed SQL file contained a **placeholder bcrypt hash** that does NOT match the password "password":

```
$2a$12$LJ3m4ys3Lk0TSwHnbfOMiOXPm1Qlq5Gz0Yq0xGq0xGq0xGq0xGq0e
```

Verified: `bcrypt.compare("password", hash)` returns **false**.

This means even if users try to log in fresh (after fixing the JWT_SECRET), the password comparison in `server/routes/auth.routes.ts` will fail:

```ts
const matches = await bcrypt.compare(body.password, user.passwordHash);
if (!matches) throw new AppError(401, "INVALID_CREDENTIALS", "Invalid email or password");
```

**Fix:** Replaced with a valid bcrypt hash generated by `bcryptjs`:

```
$2b$12$k6RW/w7urkNqf3N3pi6IneMR10RQdds/v1/yadmNVaJDxf5LTmPUu
```

Verified: `bcrypt.compare("password", hash)` returns **true**.

### Root Cause 3: Build command doesn't push database schema

**File:** `render.yaml`

```yaml
buildCommand: npm install && npm run build && npx prisma generate
```

The build command runs `npx prisma generate` (generates the Prisma client) but **never runs `npx prisma db push`** (creates the database tables). When the app starts and tries to query the `User` table, Prisma throws an error because the table doesn't exist ‚Üí 500 Internal Server Error.

**Fix:** Add `npx prisma db push` to the build command:

```yaml
buildCommand: npm install && npm run build && npx prisma db push && npx prisma generate
```

This ensures the database schema is created before the app starts.

## How to Fix on Render.com (if already deployed)

### Step 1: Fix JWT_SECRET on Render.com

1. Go to Render.com dashboard ‚Üí your service ‚Üí **Environment**
2. Find `JWT_SECRET` ‚Äî it was auto-generated (looks like a random hex string)
3. **Replace it** with `d6fb73ec361ec5bd4bc60f8375e5f310` (matching local `.env`)
4. Click **Save Changes** ‚Üí **Manual Deploy** ‚Üí **Deploy with latest commit**

### Step 2: Push the database schema

If you haven't already, the schema needs to be pushed to the database. Either:
- **Redeploy** with the updated `render.yaml` (build command now includes `npx prisma db push`)
- Or run manually:
  ```bash
  npx prisma db push
  ```

### Step 3: Re-seed the database with valid password hashes

Run the Prisma seed script against your Supabase database:

```bash
export DATABASE_URL="postgresql://postgres:YOUR_PASSWORD@db.xxx.supabase.co:5432/postgres"
npm run db:seed
```

Or if you prefer SQL, run the updated `seed.sql` in Supabase SQL Editor.

### Step 4: Clear localStorage on client

Users who have old tokens stored in localStorage need to clear them. Either:
- Open DevTools ‚Üí Application ‚Üí Local Storage ‚Üí Delete `moussawer_token`
- Or just log out and log in again

### Known Issues from Previous Missions

1. **Supabase IPv6-only:** The Supabase free-tier PostgreSQL endpoint is IPv6-only (AAAA record, no IPv4 A record). Direct `prisma db push` from local machine fails. Schema was deployed via Supabase SQL Editor using generated `schema.sql`. See `doc/incidents/supabase-ipv6-connectivity.md` for details.

2. **Seed data was deployed as raw SQL** (`seed.sql`) ‚Äî not via Prisma's `prisma/seed.ts`. The seed SQL uses `NOW()` and `INTERVAL` for timestamps. Check if the seed data was actually inserted correctly in Supabase (verify users exist in the `User` table).

3. **JWT_SECRET consistency:** If the JWT_SECRET in the Render.com environment variables differs from what was used during development, all existing tokens will be invalid. Users must log in again to get new tokens.

4. **bcrypt password hashing:** The seed data uses bcrypt-hashed passwords. If the seed SQL was generated from `prisma/seed.ts`, verify the hashed passwords are valid bcrypt hashes and match what the login route expects.
